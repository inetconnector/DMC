# How DMC's Large Context Works

Language: [Deutsch](GROSSER_KONTEXT.md) | **English**

This document explains the large-context architecture of the DMC Android app.
It is intended to let any programmer understand the idea, the actual runtime
flow, and its limitations without having to read the entire C++ implementation
first.

## Short Version

A language model does not process text directly as words. It processes
**tokens**. For every token it has already processed, llama.cpp keeps computed
intermediate data in a **KV cache**. A large KV cache consumes a large amount
of memory. On a phone, a dense KV cache for 131,072 tokens can be too large,
too slow, or impossible to allocate reliably, depending on the model.

DMC therefore separates two things:

1. The **logical history** continues to hold the complete sequence of chat
   token IDs.
2. The **physical context** contains only the tokens for which llama.cpp has
   currently built a KV cache on the device.

As long as everything fits into the physical context, the app uses the normal
dense path without any selection. Only when space becomes tight does DMC make
a deterministic selection consisting of:

- the beginning, including the system instruction and chat prefix,
- the newest tokens,
- several older regions at progressively larger temporal distances.

The selection is sorted chronologically and passed through the real model
again. This builds a new, smaller KV cache. The complete logical token history
is preserved and remains available for later rebuilds.

## What Does "Context" Mean?

Context is everything the model can consider when predicting the next token:
the system instruction, user messages, assistant responses, and any processed
attachments.

Three sizes must be distinguished:

| Term | Meaning |
| --- | --- |
| Training context | The context size reported in the GGUF model metadata. |
| Logical context | The complete token history managed by DMC and the context size reported to the UI. |
| Physical context | The llama.cpp KV window actually allocated in device memory. |

The currently used Gemma model reports a training context of 131,072 tokens.
The Android app therefore reports 131,072 as its logical window, while it
normally allocates only 16,384 physical context positions on the phone. If
that allocation fails, the runtime tries progressively smaller windows, for
example 8,192 and finally 4,096 tokens.

Important: displaying `131072` does not mean that a dense 131K KV cache is in
RAM at all times. It means that DMC manages a logical history of that scale and
uses it to build a smaller working context. How useful 131K tokens are also
depends on the context length and long-context behavior for which the model
itself was trained.

## Why the KV Cache Uses So Much Memory

For autoregressive Transformer models, a key and value state is stored for
every previous token across many model layers. This prevents the model from
having to recompute the entire sequence for every new token.

The memory requirement grows approximately in proportion to:

```text
token count x layer count x KV dimension x data type size
```

Quantizing GGUF model weights, for example to Q4, primarily reduces the model
weights. It does not automatically make every KV state equally small. A model
can therefore fit and run on a phone while a fully dense 131K context would
still require too much RAM.

## DMC's Two Storage Areas

### 1. Logical Token History

`g_dmc_token_history` contains the canonical sequence of token IDs. It
includes:

- the formatted system message,
- the formatted user messages,
- response tokens generated by the model,
- model end tokens such as EOG.

This history stores token IDs, not the large KV state for every token. It is
therefore much smaller than a KV cache of the same length. DMC does not
recursively summarize the text or replace it with an automatically generated
summary.

### 2. Physical llama.cpp Context

The physical context is the normal llama.cpp context and its KV cache. On
Android, the target size is 16,384 tokens. Four positions are withheld as
technical overflow headroom. The runtime also reserves space for the next
response.

Without an explicit output limit, 4,096 positions are reserved for
continuation by default. With an explicit limit, at least 512 positions are
reserved. The reserve is capped at half of the physical window.

## When DMC Starts Selecting

A short chat follows the normal dense path:

```text
new tokens -> llama_decode -> KV cache grows
```

DMC rebuilds the context only when both the logical history and the current
physical state exceed the available prompt budget. After a rebuild, the
selected KV state can be reused for subsequent short messages. The entire
selection is therefore not replayed for every single turn.

The simplified flow is:

```text
complete token history
          |
          v
fits into physical prompt budget? ----- yes ---> continue normally
          |
          no
          |
          v
DMC selects beginning + newest region + older replay regions
          |
          v
deduplicate and sort the selection chronologically
          |
          v
clear the physical KV cache
          |
          v
replay the selected original tokens through llama_decode
          |
          v
continue generation normally, one token at a time
```

## Multiresolution Selection

The Android configuration is created by `runtime_dmc_config()`:

| Parameter | Android value | Purpose |
| --- | ---: | --- |
| `block_size` | 64 | Base size of an older replay block. |
| `local_window` | 2,048 | Newest tokens that are preserved in full. |
| `global_tokens` | at least 64, at most 1,024 | Start of the sequence, especially the system and formatting prefix. |
| `replay_levels` | 7 | Number of possible older time scales. |

The number of global tokens depends on the size of the system prefix:

```text
global_tokens = max(64, min(system token count, 1024))
```

For each replay level `L`, the simplified formulas are:

```text
distance(L)   = max(local_window, block_size) x 2^L
block_size(L) = block_size x 2^L
```

Nearby replay regions are small and detailed. More distant regions are larger
but are spaced farther apart. This is the meaning of **Deterministic
Multiresolution Context**: deterministic context represented at multiple
temporal resolutions.

All selected token positions are combined, duplicate positions are removed,
and the result is sorted in ascending order. Chronological order matters
because a causal decoder must replay tokens from oldest to newest.

If a selection is still larger than the physical budget, the global beginning
and local ending take priority. Remaining positions are filled with replay
tokens in deterministic order. The newest token must always remain in the
selection; otherwise the plan is treated as an error.

## Concrete Example: 32K Logical, 4K Physical

The native self-test plans 32,768 logical tokens for a budget of 4,096 physical
tokens. With a 64-token global region, the result is:

| Region | Token positions | Count |
| --- | --- | ---: |
| Global beginning | 0 through 63 | 64 |
| Local ending | 30,720 through 32,767 | 2,048 |
| Replay 1 | 30,656 through 30,719 | 64 |
| Replay 2 | 28,544 through 28,671 | 128 |
| Replay 3 | 24,320 through 24,575 | 256 |
| Replay 4 | 15,872 through 16,383 | 512 |
| **Total** | after deduplication | **3,072** |

The result stays below the 4K budget and contains the beginning, the complete
newest region, and several older time scales. The same history and
configuration always produce exactly the same selection.

## What Exactly Happens During a Rebuild?

`rebuild_dmc_context()` performs these steps:

1. `plan_runtime_context()` receives the length of the logical history and the
   physical token budget.
2. The plan returns positions in the complete history.
3. The runtime reads the unchanged original token IDs at those positions.
4. The existing llama.cpp KV cache is cleared.
5. The selected tokens are processed again with `llama_decode()` in batches of
   up to 512 tokens.
6. llama.cpp computes a normal new KV cache from them.
7. The complete logical history remains stored separately.

"Exact attention over selected spans" means that no approximate attention
value is invented for the tokens that were selected. The Android runtime lets
the real model evaluate these tokens again. Tokens that were not selected are,
however, not directly visible in the physical KV cache created by that rebuild.

The Python and C++ references also contain an explicit `exact_attention()`
function. Tests show that it matches dense attention when all tokens are
selected. Android does not replace llama.cpp's attention kernel with this
reference function. Android uses the shared selection plan and rehydrates the
normal llama.cpp context from that plan.

## Long Responses Are Not Cut Off at the Window Boundary

The physical KV cache also grows while a response is generated. Every
generated token is appended to the logical history at the same time. When the
physical context reaches its limit, the runtime performs another DMC rebuild
and reserves another 4,096 positions for continuation.

Generation then continues until:

- the model produces an end token itself, or
- an output-token limit explicitly supplied by the caller is reached.

A full physical position is therefore no longer treated incorrectly as a
normal end of the answer. If a required rebuild fails, the runtime reports an
error instead of presenting a visibly incomplete response as complete.

## What DMC Deliberately Does Not Do

In this implementation, DMC is:

- **not recursive summarization** of older messages,
- **not semantic vector search**,
- **not a RAG system**,
- **not an embedding or ANN index**,
- **not learned routing**,
- **not content-dependent top-k selection**,
- **not fully dense 131K attention on a phone**.

Selection depends only on positions, fixed parameters, and the available
budget. This makes its behavior reproducible, testable, and comparatively
straightforward to port.

## Benefits and Limitations

### Benefits

- A much smaller physical KV memory requirement than a dense 131K window.
- Short chats continue to use the unchanged dense path.
- The beginning and newest part of the conversation receive priority.
- Older time ranges are represented at multiple resolutions.
- The same inputs and parameters produce the same selection.
- No additional summarization request or second model is required.
- The complete token sequence remains available for later rebuilds.

### Limitations

- Old tokens that are not selected cannot be attended to directly after the
  current rebuild. A detail located in a gap can therefore be missed.
- A rebuild costs compute time because selected tokens must pass through the
  model again.
- Selected tokens are replayed densely and chronologically. Their original
  large distances are therefore not represented identically in the physical
  context.
- The training context reported by the model remains an important quality
  boundary. DMC does not automatically turn a short-context model into a good
  long-context model.
- A displayed logical context window is not proof of perfect recall of every
  individual token.
- Quality must be measured with real long-context tasks and regression tests.
  Determinism alone does not guarantee a good answer.

## Self-Test, Build Gate, and Diagnostics

A native self-test runs whenever a model is prepared. Among other things, it
checks that:

- a 32K history is actually compacted,
- two identical plans produce the same token positions,
- the result does not exceed the physical budget,
- the newest token is preserved.

The Android build also searches the packaged ARM64 library for DMC markers.
This is intended to prevent publishing an APK that does not contain the active
DMC runtime.

Important Logcat messages are:

```text
DMC_RUNTIME self-test passed
DMC_RUNTIME enabled=1 physical_context=... logical_context=...
DMC_RUNTIME incremental turn: mode=dense ...
DMC_RUNTIME incremental turn: mode=selected-reuse ...
DMC_RUNTIME rebuild=... logical=... selected=... spans=...
DMC_RUNTIME generation continuation: ...
```

`mode=dense` means that no selection has been necessary yet.
`mode=selected-reuse` means that an already compacted physical context is being
reused. A `rebuild` message identifies an actual DMC rebuild.

| `scripts/windows/verify-android-dmc.ps1` | Verifies DMC markers in the built Android library or APK. |

## Rules for Safe Changes

Anyone changing DMC should preserve at least these invariants:

1. Short histories remain complete and dense.
2. Tokens from the future are never selected.
3. The newest token is always included.
4. Selection never exceeds the physical budget.
5. Replay order is chronological.
6. The same input produces the same selection.
7. The canonical token history is not lost during a KV rebuild.
8. A full KV position must not silently truncate a response.
9. Rebuild failures must be reported visibly.
10. The Python reference, C++ reference, and Android behavior remain aligned.

After a change, at least the Python tests, C++ tests, Android DMC marker check,
and a real long-context test on a device should be run.

## One-Sentence Summary

DMC keeps the complete chat as a small logical token sequence, while the phone
uses a limited real llama.cpp KV cache that is rebuilt when needed from the
beginning, the newest text, and deterministic older replay regions.
